# AI Agent 的高效上下文工程

> 原文：[Effective context engineering for AI agents](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)
> 作者：Prithvi Rajasekaran, Ethan Dixon, Carly Ryan, Jeremy Hadfield
> 发布时间：2025 年 9 月 29 日

**核心观点**：上下文是 AI Agent 的关键但有限的资源。本文探讨了有效策划和管理上下文以驱动 Agent 的策略。

---

经过几年提示工程（prompt engineering）成为应用 AI 领域关注的焦点，一个新术语开始崭露头角：**上下文工程（context engineering）**。使用语言模型进行构建，越来越多地不再是寻找提示中正确的词语和短语，而是回答一个更广泛的问题：**"什么样的上下文配置最有可能生成我们期望的模型行为？"**

上下文（Context）指的是从大语言模型（LLM）采样时包含的 token 集合。工程上的挑战在于，针对 LLM 固有的约束，优化这些 token 的效用，以持续达到期望的结果。有效驾驭 LLM 通常需要以上下文的方式思考——换句话说，就是考虑 LLM 在任何给定时刻可用的整体状态，以及该状态可能产生的潜在行为。

在本文中，我们将探索新兴的上下文工程艺术，并为构建可控、高效的 Agent 提供一套精炼的心智模型。

---

## 上下文工程 vs. 提示工程

在 Anthropic，我们认为上下文工程是提示工程的自然演进。

- **提示工程**是指编写和组织 LLM 指令以获得最佳结果的方法（参见我们的[文档](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/overview)以获取概述和有用的提示工程策略）。
- **上下文工程**是指在 LLM 推理期间策划和维护最佳 token（信息）集的策略，包括提示之外可能出现在上下文中的所有其他信息。

在早期使用 LLM 进行工程构建时，提示是 AI 工程工作中最大的组成部分，因为大多数日常聊天交互之外的使用场景都需要优化的提示来完成一次性分类或文本生成任务。顾名思义，提示工程的主要焦点是如何编写有效的提示，特别是系统提示。

然而，随着我们转向构建能够在多轮推理和更长的时间跨度上运行的更强大的 Agent，我们需要管理整个上下文状态的策略——包括系统指令、工具、模型上下文协议（MCP）、外部数据、消息历史等。

在循环中运行的 Agent 会产生越来越多的可能对下一轮推理相关的数据，这些信息必须循环精炼。上下文工程就是从不断演变的信息宇宙中，精心策划哪些信息会被放入有限的上下文窗口的艺术和科学。

与编写提示这一离散任务不同，上下文工程是迭代的，其策划阶段在每次我们决定向模型传递什么内容时都会发生。

## 为什么上下文工程对构建能力强大的 Agent 如此重要

尽管 LLM 速度很快，并且能够管理越来越大规模的数据，但我们观察到，LLM 和人类一样，在某个临界点后会**失去注意力或产生混乱**。针对"大海捞针"式基准测试的研究发现了**上下文腐烂（context rot）** 的概念：随着上下文窗口中的 token 数量增加，模型从该上下文中准确回忆信息的能力会下降。

虽然某些模型表现出更温和的退化，但这种特征在所有模型中都会出现。因此，上下文必须被视为一种**边际收益递减**的有限资源。与人类的工作记忆容量有限一样，LLM 在解析大量上下文时也有一个"注意力预算"。每引入一个新 token 就会消耗一定量的预算，这就更需要精心策划可供 LLM 使用的 token。

这种**注意力稀缺**源于 LLM 的架构限制。LLM 基于 Transformer 架构，该架构使每个 token 能够在整个上下文中关注其他每个 token。对于 n 个 token，这意味着 n² 个两两关系。

随着上下文长度的增加，模型捕捉这些两两关系的能力被拉得更薄，在上下文大小和注意力聚焦之间形成了天然张力。此外，模型的注意力模式来自训练数据分布，其中较短的序列通常比长序列更常见。这意味着模型在处理全上下文依赖关系时的经验和专门参数都较少。

位置编码插值（Position encoding interpolation）等技术允许模型通过将长序列自适应回原本训练时使用的较短上下文来处理更长的序列，尽管在 token 位置理解上会有些退化。这些因素形成了一个**性能梯度**而非硬性悬崖：模型在更长的上下文中仍然保持高能力，但在信息检索和长程推理的精确度上可能不如在短上下文中的表现。

这些现实意味着，**深思熟虑的上下文工程对于构建能力强大的 Agent 至关重要。**

## 高效上下文的解剖结构

鉴于 LLM 受限于有限的注意力预算，好的上下文工程意味着找到**最小的高信号 token 集**，以最大化某些期望结果的可能性。实践这一点说起来容易做起来难，但在下一节中，我们将概述这一指导原则在上下文各组件中的实际含义。

### 系统提示

系统提示应该**极其清晰**，使用**简单直接的语言**，以**恰当的高度**为 Agent 呈现思路。恰当的高度是在两种常见失败模式之间的"黄金地带"：

- 一端：工程师在提示中硬编码复杂、脆弱的逻辑来强制 Agent 的特定行为——这创造了脆弱性并增加了维护复杂度。
- 另一端：工程师有时提供模糊、高层的指导，未能给 LLM 提供期望输出的具体信号，或错误地假设了共享上下文。

最佳高度在于平衡：既要足够具体以有效引导行为，又要足够灵活，为模型提供强大的启发式指引。

> **图：系统提示的"黄金地带"**——在"脆弱的硬编码 if-else 提示"和"过于通用的提示"之间取得平衡，找到既足够具体又足够灵活的最佳高度。

我们建议将提示组织成不同的部分（如 `<background_information>`、`<instructions>`、`## 工具指导`、`## 输出描述` 等），并使用 XML 标签或 Markdown 标题等技巧来划分这些部分——尽管随着模型能力的增强，提示的确切格式可能变得不那么重要。

无论如何组织系统提示，你都应该追求能**完整概述期望行为的最小信息集**。（注意：最小并不一定意味着短；你仍然需要在前端给 Agent 足够的信息，以确保它遵循期望行为。）最好先用**最小的提示**和最强大的模型进行测试，观察它在任务上的表现，然后根据测试中发现的失败模式，**添加清晰的指令和示例**来改进性能。

### 工具

工具允许 Agent 与其环境交互，并在工作中拉取新的额外上下文。由于工具定义了 Agent 与其信息/行动空间之间的契约，因此**工具必须促进效率**——既要返回 token 高效的信息，又要鼓励高效的 Agent 行为。

在我们的[用 AI Agent 为 AI Agent 编写工具](https://www.anthropic.com/engineering/writing-tools-for-ai-agents-with-ai-agents)一文中，我们讨论了如何构建 LLM 容易理解且功能重叠最小的工具。与设计良好的代码库中的函数类似，工具应该是自包含的、对错误鲁棒、并且对其预期用途极其清晰的。输入参数同样应该具有**描述性、无歧义**，并发挥模型的固有优势。

我们观察到最常见的失败模式之一是**工具集臃肿**——覆盖太多功能，或在选择哪个工具时产生歧义决策点。如果人类工程师都无法明确说出在特定情况下应该使用哪个工具，就不能指望 AI Agent 做得更好。正如我们后面将讨论的，**为 Agent 策划最小可行工具集**也有助于在长交互过程中实现更可靠的上下文维护和剪枝。

### 示例

提供示例，也称为**少样本提示（few-shot prompting）**，是一个广为知名的、我们继续强烈推荐的最佳实践。然而，团队常常会在一份提示中塞满一长串边缘案例，试图阐明模型应遵循的每条规则。**我们不推荐这样做。**

相反，我们建议精心策划一组**多样化、规范的示例**，有效展示 Agent 的期望行为。对 LLM 来说，示例就是"一图胜千言"中的"图"。

我们在上下文各组件（系统提示、工具、示例、消息历史等）上的总体指导是：**精心思考，保持上下文信息丰富但精炼。** 接下来，让我们深入探讨在运行时动态检索上下文的方法。

## 上下文检索与 Agent 搜索

在[构建高效的 AI Agent](https://www.anthropic.com/engineering/building-effective-agents)一文中，我们强调了基于 LLM 的工作流与 Agent 之间的区别。自那篇文章之后，我们倾向于一个简单的 Agent 定义：**LLM 在循环中自主使用工具。**

与客户合作的过程中，我们看到业界正在向这个简单的范式收敛。随着底层模型能力的增强，Agent 的自主程度也可以扩展：更智能的模型使 Agent 能够独立导航复杂的问题空间并从错误中恢复。

现在我们看到工程师在设计 Agent 上下文的方式上正在发生转变。如今，许多 AI 原生应用使用某种形式的**基于嵌入的推理前检索**来为 Agent 提供重要上下文供其推理。随着领域向更 Agent 化的方法过渡，我们越来越多地看到团队用**"即时"（just-in-time）上下文策略**来增强这些检索系统。

基于"即时"方法构建的 Agent 不会预先处理所有相关数据，而是维护**轻量级标识符**（文件路径、存储的查询、网络链接等）。

![推理前检索与即时上下文检索的对比](images/context-engineering-hero.png)

这种范式差异对我们如何构建系统有显著影响：

- **推理前检索**：你在推理前将知识注入到系统提示中——所有重工作在推理前完成，LLM 一次性消费所有内容。
- **即时上下文检索**：Agent 在推理过程中自主决定何时需要更多信息，然后自行启动工具调用来获取它。Agent 维护的是*通往*信息的指针，而不是信息本身。数据只有在需要时才被拉取，从而保持上下文的轻量和集中。

这种转变类似于实时视频流与从文件服务器下载的对比：解析一个 10,000 token 的网页来找到两个相关的句子，比在 Agent 运行时将完整的 10,000 token 网页填入系统提示要高效得多。

**那么，该如何决定使用哪种方法呢？**

最佳方法取决于你的任务约束。如果你的 Agent 每次都能提前知道需要的信息，那么推理前检索可能是合适的（例如，每天阅读相同法规的合规 Agent）。使用"即时"方法，Agent 可能会浪费 token 去寻找它每次都能轻易获得的相同信息。

然而，如果你的 Agent 需要根据用户提示灵活地从多个来源搜索信息，"即时"检索提供了更高的 token 效率和情境灵活性。当 Agent 是自主的且目标是通用的（能在许多不同任务中有效）时，"即时"方法通过仅在需要时才拉取相关信息，使 Agent 能最大限度地利用其注意力预算。

**"即时"检索是一种日益普遍的上下文工程模式，它利用 Agent 的工具使用能力，在推理过程中按需获取信息，而不是预先加载所有内容。**

## Agent 上下文压缩

尽管即时检索使 Agent 能够仅在需要时获取信息，但在长时间运行的轨道上，内部推理数据（消息历史、工具调用等）会积累。上下文压缩（Compaction）是指**消除或概括上下文窗口中的内容以减少 token 数量**的行为。

这是上下文工程中一个重要的困境：当每个新 token 的增加都面临递减的边际收益，且 token 总成本很高时，我们不能无限增加消息历史的容量而期望获得更好的 Agent 行为。然而，过去的相关信息对帮助 Agent 理解状态和做出决策至关重要。

关键是在**忘记太多**（重新引入幻觉和协调失败）和**保留太多**之间取得平衡。

### 消息摘要

一种常见的压缩技术是在将旧消息纳入上下文窗口之前对其进行摘要。保留一个正在进行的交互的运行摘要，可以显著减少上下文消耗并提高响应质量，尤其是在连续任务中。

我们发现一个简单的特定配方很有效：如果 Agent 具有读写任意文件的能力，只需要简单地维护一个包含当前交互最新摘要的 `scratchpad.md` 文件，就像 Claude Code 所做的那样。

### 工具结果清理

所有压缩技术中最简单的一种是工具结果清理：一旦工具结果被使用，就从上下文窗口中清除它。作为工程师，我们经常向 Agent 提供它们核心任务不需要的数据（例如，大型日志文件、长时间跟踪的原始输出、大型搜索结果集）。在设计工具时，我们建议确保工具返回 LLM 所需的最小数据量，而不是返回完整的结果并让 LLM 自行提取。

与其猜测 Agent 可能需要什么，更好的方法是从完整的结果开始，从跟踪中检查哪些部分被实际使用，然后迭代改进，消除多余的内容。

一个低挂果实式的多余内容是**清理工具调用和结果**——一旦一个工具在消息历史深处被调用过，Agent 为什么还需要再次看到原始结果？一种最安全、最轻量的压缩形式是工具结果清理，最近作为 Claude Developer Platform 的一项功能推出。

### 结构化笔记（Agent 记忆）

结构化笔记，或称 **Agent 记忆（agentic memory）**，是一种技术，Agent 定期将笔记写入持久化存储在上下文窗口之外的内存中。这些笔记在后续时间被拉回上下文窗口。

这种策略提供了开销最小的持久记忆。就像 Claude Code 创建待办事项列表，或你的自定义 Agent 维护 `NOTES.md` 文件一样，这种简单的模式允许 Agent 在复杂任务中跟踪进展，维护原本会在数十次工具调用中丢失的关键上下文和依赖。

[Claude 玩 Pokémon](https://www.anthropic.com/engineering/claude-pokemon)展示了记忆如何在非编码领域改变 Agent 能力。Agent 在数千个游戏步骤中维持精确的记录——追踪目标，比如"在过去的 1,234 步中，我一直在 1 号道路上训练我的宝可梦，皮卡丘已经提升了 8 级，目标 10 级"。在没有关于记忆结构的任何提示的情况下，它会自发发展出已探索区域的地图，记住已解锁的关键成就，并维护战斗策略的战略笔记，帮助它学习针对不同对手最有效的攻击方式。

在上下文重置后，Agent 读取自己的笔记并继续多小时的训练序列或地牢探索。这种跨摘要步骤的一致性使得长期策略成为可能，而如果将所有信息保存在 LLM 的上下文窗口中是不可能实现的。

作为 Sonnet 4.5 发布的一部分，我们在 Claude Developer Platform 上以公开测试版形式发布了**记忆工具**，通过文件系统支持更容易地在上下文窗口外存储和查阅信息。这使得 Agent 可以随时间构建知识库，跨会话维护项目状态，并在不将所有内容保留在上下文中的情况下引用之前的工作。

### 子 Agent 架构

子 Agent 架构提供了另一种绕开上下文限制的方式。不是由一个 Agent 试图维护整个项目的状态，而是**专门的子 Agent**可以用干净的上下文窗口处理聚焦的任务。主 Agent 通过高层计划进行协调，而子 Agent 执行深度的技术工作或使用工具查找相关信息。每个子 Agent 可能进行大量探索，使用数万 token 甚至更多，但只返回压缩、蒸馏后的摘要（通常 1,000-2,000 token）。

这种方法实现了清晰的关注点分离——详细的搜索上下文保留在子 Agent 内部，而主导 Agent 专注于综合和分析结果。这种模式在[我们如何构建多 Agent 研究系统](https://www.anthropic.com/engineering/building-multi-agent-research-system)一文中展示过，它在复杂研究任务上相比于单 Agent 系统有显著改进。

### 如何选择

这些方法之间的选择取决于任务特征：

- **压缩（Compaction）**：适合需要大量来回交互以保持对话流的任务；
- **结构化笔记（Note-taking）**：适合具有明确里程碑的迭代开发；
- **多 Agent 架构（Multi-agent）**：适合复杂研究和分析，其中并行探索能带来更大收益。

即使模型持续改进，在扩展交互中保持一致性的挑战仍将是构建更高效 Agent 的核心问题。

## 结论

上下文工程代表了我们在构建 LLM 时的**根本性转变**。随着模型能力的增强，挑战不再仅仅是编写完美的提示——而是在每一步**精心策划什么信息进入模型有限的注意力预算**。

无论你是在为长周期任务实现压缩、设计 token 高效的工具，还是让 Agent 即时探索其环境，指导原则始终如一：**找到最小的高信号 token 集，以最大化期望结果的可能性。**

我们概述的这些技术将随着模型的改进而持续演化。我们已经看到更智能的模型需要更少的规范性工程，允许 Agent 以更高的自主性运行。但即使能力在扩展，**将上下文视为珍贵的有限资源**仍将是构建可靠、高效 Agent 的核心。
