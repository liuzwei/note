# 扩展托管 Agent：将大脑与双手解耦

> 原文：[Scaling Managed Agents: Decoupling the brain from the hands](https://www.anthropic.com/engineering/managed-agents)
> 作者：Lance Martin, Gabe Cemaj, Michael Cohen
> 发布时间：2026 年 4 月 8 日

**核心观点**：Harness（框架）编码了对模型能力的假设，这些假设会随着模型升级而过时。Managed Agents——我们用于长时间运行 Agent 任务的托管服务——围绕稳定的接口构建，这些接口在 harness 变化时保持稳定。

> 开始使用 Claude Managed Agents，请参阅我们的[文档](https://platform.claude.com/docs/en/agent-sdk/overview)。

工程博客的一个持续话题是如何构建高效的 Agent 以及如何为长时间运行的工作设计 harness（框架）。这些工作的共同点是：**harness 编码了关于 Claude 无法独立完成哪些事情的假设**。然而，这些假设需要经常被质疑，因为它们可能随着模型的改进而过时。

举一个例子：在之前的工作中，我们发现 Claude Sonnet 4.5 会在感知到上下文窗口接近上限时过早地结束任务——这种行为有时被称为"上下文焦虑"（context anxiety）。我们通过在 harness 中添加上下文重置来解决这个问题。但当我们在 Claude Opus 4.5 上使用同一个 harness 时，我们发现这种行为已经消失了。重置操作变成了死重。

我们预计 harness 会继续进化。因此，我们构建了 **Managed Agents**：Claude Platform 平台上的一个托管服务，它通过一套旨在超越任何特定实现（包括我们当前运行的实现）的接口，为你运行长时间运行的 Agent。

构建 Managed Agents 意味着解决计算领域的一个老问题：**如何为"尚不可想象的程序"设计系统**。几十年前，操作系统通过将硬件虚拟化为抽象层——进程、文件——来解决这个问题，这些抽象层的通用性足以容纳尚未存在的程序。抽象层比硬件更持久。`read()` 命令既不知道也不关心它访问的是 1970 年代的磁盘还是现代的 SSD。上层的抽象保持稳定，而下层的实现可以自由变化。

Managed Agents 遵循相同的模式。我们将 Agent 的组件虚拟化：

- **Session（会话）**：所有发生的事情的仅追加日志
- **Harness（框架）**：调用 Claude 并将 Claude 的工具调用路由到相关基础设施的循环
- **Sandbox（沙箱）**：Claude 可以运行代码和编辑文件的执行环境

这使得每个组件的实现可以在不影响其他组件的情况下进行替换。**我们坚持的是这些接口的形状，而不是它们背后运行什么。**

## 不要养宠物

我们最初将所有 Agent 组件放入一个容器中，这意味着 session、agent harness 和 sandbox 共享同一个环境。这种方法的好处包括：文件编辑是直接的系统调用，不需要设计服务边界。

但通过将所有组件耦合在一个容器中，我们遇到了一个古老的基础设施问题：**我们养了一只宠物**。在"宠物 vs 牲畜"的类比中，宠物是你命名并亲手照料、不能失去的个体，而牲畜是可互换的。在我们的案例中，服务器成了那只宠物；如果容器失败，session 就丢失了。如果容器无响应，我们得把它"护理"回健康状态。

护理容器意味着调试无响应的卡死 session。我们唯一的窗口是 WebSocket 事件流，但它无法告诉我们失败发生在哪里——这意味着 **harness 中的 bug、事件流中的数据包丢失、或者容器离线，都表现为相同的现象**。为了找出问题所在，工程师必须打开容器的 shell，但因为该容器通常也包含用户数据，这种方法本质上意味着我们缺乏调试能力。

第二个问题是，harness 假设 Claude 处理的所有内容都存在于同一容器中。当客户要求我们将 Claude 连接到他们的虚拟私有云（VPC）时，他们要么必须将他们的网络与我们对等连接，要么在自己的环境中运行我们的 harness。**构建在 harness 中的假设，在我们想将其连接到不同基础设施时成了问题。**

## 将大脑与双手解耦

我们找到的解决方案是将我们认为是"大脑"的部分（Claude 及其 harness）与"双手"（执行操作的 sandbox 和工具）以及"会话"（会话事件的日志）解耦。每一个都成为一个对其他部分很少做假设的接口，每个都可以独立地失败或被替换。

### Harness 离开容器

将大脑与双手解耦意味着 harness 不再存在于容器内部。它调用容器的方式与调用任何其他工具一样：`execute(name, input) → string`。容器变成了"牲畜"。如果容器挂掉，harness 会以工具调用错误的形式捕获失败，并将错误传回给 Claude。如果 Claude 决定重试，新的容器可以通过标准配方重新初始化：`provision({resources})`。我们不再需要将失败的容器"护理"回健康状态。

### 从 Harness 故障中恢复

Harness 也变成了"牲畜"。因为 session 日志存在于 harness 之外，harness 中没有任何内容需要从崩溃中存活。当一个 harness 失败时，新的 harness 可以通过 `wake(sessionId)` 重启，使用 `getSession(id)` 取回事件日志，并从最后一个事件继续执行。在 Agent 循环期间，harness 通过 `emitEvent(id, event)` 向 session 写入，以保持事件的持久记录。

### 安全边界

在耦合设计中，Claude 生成的任何不受信任的代码都在同一容器中运行，与凭证共存——因此**提示注入只需要说服 Claude 读取自己的环境**。一旦攻击者获得了这些令牌，他们就可以生成全新的、不受限制的 session 并将工作委托给它们。窄范围化（narrow scoping）是一种明显的缓解措施，但这编码了对 Claude 无法使用有限令牌做什么的假设——而 Claude 正变得越来越聪明。结构性的修复是**确保令牌永远无法从 Claude 生成代码运行的 sandbox 中访问到**。

我们使用两种模式来确保这一点。认证可以绑定到资源上，或者保存在 sandbox 外部的保险库中：

- **对于 Git**：我们在 sandbox 初始化期间使用每个仓库的访问令牌来克隆仓库，并将其连接到本地 git remote。Git push 和 pull 从 sandbox 内部工作，而 Agent 从未接触令牌本身。
- **对于自定义工具**：我们支持 MCP 并将 OAuth 令牌存储在安全的保险库中。Claude 通过专用的代理调用 MCP 工具；这个代理接收与 session 关联的令牌，然后从保险库获取相应的凭证，并调用外部服务。**Harness 永远不会感知到任何凭证。**

## Session 不是 Claude 的上下文窗口

长时间运行的任务通常会超出 Claude 上下文窗口的长度，而解决这一问题的标准方法都涉及关于保留什么内容的不可逆决策。我们在之前关于上下文工程的工作中探讨过这些技术。例如：

- **压缩（Compaction）**：让 Claude 保存其上下文窗口的摘要
- **记忆工具（Memory tool）**：让 Claude 将上下文写入文件，从而实现在 session 间学习
- **上下文修剪（Context trimming）**：有选择地移除令牌，如旧的工具结果或思考块

但**有选择地保留或丢弃上下文的不可逆决策可能导致失败**。很难知道未来的回合需要哪些令牌。如果消息通过压缩步骤进行了转换，harness 会从 Claude 的上下文窗口中移除压缩过的消息，而这些消息只有被存储才能恢复。

之前的工作探索了通过将上下文存储为上下文窗口外部的对象来解决这个问题的方法。例如，上下文可以是 REPL 中的一个对象，LLM 通过编写代码来过滤或切片它以编程方式访问。

在 Managed Agents 中，会话提供了同样的好处，充当了存在于 Claude 上下文窗口外部的上下文对象。但上下文不是存储在 sandbox 或 REPL 中，而是持久地存储在 session 日志中。`getEvents()` 这个接口允许"大脑"通过选择事件流的位置切片来查询上下文。该接口可以灵活使用，允许"大脑"从上次停止阅读的地方继续、回退到特定时刻之前的一些事件来查看前因、或在特定操作之前重新阅读上下文。

任何获取的事件也可以在传入 Claude 的上下文窗口之前在 harness 中进行转换。这些转换可以是 harness 编码的任何内容，包括实现高提示缓存命中率的上下文组织和上下文工程。我们将会话中的可恢复上下文存储和 harness 中的任意上下文管理这两个关注点分离开来，因为我们无法预测未来模型需要什么样的特定上下文工程。接口将上下文管理推给 harness，只保证 session 是持久的且可供查询。

## 多大脑，多双手

### 多大脑（Many brains）

将大脑与双手解耦解决了我们最早的客户抱怨之一。当团队希望 Claude 针对他们自己 VPC 中的资源工作时，唯一的途径是将他们的网络与我们对等连接，因为承载 harness 的容器假定每个资源都在它旁边。一旦 harness 不再在容器中，这个假设就消失了。

同样的改变也带来了性能回报。当我们最初将大脑放在容器中时，意味着**多大脑就需要同样多的容器**。对于每个大脑，在容器配置完成之前无法进行推理；每个 session 都要预先支付完整的容器设置成本。每个 session——即使是从不接触 sandbox 的 session——都必须克隆仓库、启动进程、从我们的服务器获取待处理事件。

这种死时间表现为**首次令牌时间（TTFT, Time-to-First-Token）**，它衡量 session 在接受工作到产生第一个响应令牌之间的等待时间。TTFT 是用户最能感受到的延迟。

将大脑与双手解耦意味着**容器只有在需要时才通过工具调用**（`execute(name, input) → string`）由大脑配置。因此，不需要立即使用容器的 session 无需等待容器。推理可以在编排层从 session 日志中拉取待处理事件时立即开始。使用这种架构，我们的 **p50 TTFT 下降了约 60%，p95 下降了超过 90%**。扩展到多大脑只需要启动多个无状态 harness，并在需要时连接到双手。

### 多双手（Many hands）

我们还想要将每个大脑连接到多双手的能力。在实践中，这意味着 Claude 必须推理多个执行环境并决定将工作发送到哪里——这是比在单个 shell 中操作更困难的认知任务。我们从将大脑放在单个容器中开始，因为早期的模型不具备这种能力。随着智能水平的提升，单容器反而成了限制——当该容器失败时，我们丢失了大脑正在触及的每只手的状态。

将大脑与双手解耦使得每只手都成为一个工具：`execute(name, input) → string`——名称和输入进入，字符串返回。该接口支持任何自定义工具、任何 MCP 服务器以及我们自己的工具。**Harness 不知道 sandbox 是一个容器、一部手机、还是一个 Pokémon 模拟器**。而且由于没有手与任何大脑耦合，大脑之间可以互相传递手。

## 结论

我们面临的挑战是一个古老的问题：**如何为"尚不可想象的程序"设计系统**。操作系统之所以能持续数十年，是因为它们将硬件虚拟化为足够通用的抽象，足以容纳尚未存在的程序。通过 Managed Agents，我们旨在设计一个能够容纳未来围绕 Claude 的 harness、sandbox 或其他组件的系统。

Managed Agents 是一个**元框架（meta-harness）**，秉承同样的精神——对未来 Claude 需要的具体 harness 不做预设。相反，它是一个拥有通用接口的系统，允许许多不同的 harness。例如，Claude Code 就是一个出色的 harness，我们在各种任务中广泛使用它。我们还展示了特定任务的 Agent harness 在狭窄领域中的卓越表现。Managed Agents 可以容纳其中任何一种，并随着时间推移匹配 Claude 的智能水平。

元框架设计意味着**对围绕 Claude 的接口有明确的意见**：我们预期 Claude 将需要操作状态的能力（session）和执行计算的能力（sandbox）。我们还预期 Claude 将需要扩展到多大脑和多双手的能力。我们设计的接口使得这些能力可以在长时间周期内可靠且安全地运行。但我们对 Claude 将来需要的大脑或双手的数量或位置不做任何假设。
