### 解释：对问题进行规范化解释

模型在生成下一个 token 时，并不是只能“选概率最高的那个”。实际推理中，通常会结合一组 **解码参数（decoding parameters）** 来控制输出风格、稳定性和发散程度。

这部分是大模型应用里最常被误用、但又必须掌握的知识。

---

### 举例：举例说明问题，帮助理解不同参数会怎样影响输出

假设模型下一个 token 的候选概率大致是：

* `北京`：0.45
* `上海`：0.20
* `深圳`：0.10
* 其他：0.25

不同策略下结果会不同：

1. **Greedy**

   * 直接选 `北京`
   * 最稳定，但容易保守和重复

2. **Temperature 提高**

   * 分布被拉平
   * `上海`、`深圳` 被采样到的概率会上升

3. **Top-k**

   * 只在前 k 个候选里采样
   * 可以去掉尾部低质量噪声

4. **Top-p**

   * 选累计概率达到 p 的最小候选集合
   * 分布自适应，比固定 top-k 更常用

---

### 知识点：着重说明必须记住的技术点

1. **Greedy Search**

   * 每一步选最大概率 token
   * 优点是稳定、可复现
   * 缺点是容易陷入重复和局部最优

2. **Temperature**

   * 控制分布陡峭程度
   * `temperature < 1`：更保守
   * `temperature > 1`：更发散
   * `temperature = 0` 通常近似 greedy

3. **Top-k Sampling**

   * 只保留概率最高的 k 个 token 再采样
   * k 太小会过于死板，k 太大会引入噪声

4. **Top-p / Nucleus Sampling**

   * 只保留累计概率达到 `p` 的候选集合
   * 是一种动态裁剪方式
   * 工程实践里往往比单纯 top-k 更稳

5. **Repetition Penalty**

   * 对已生成 token 降权
   * 用来减少模型反复复读
   * 过大时可能导致语言不自然

6. **Length、Stop、EOS 控制**

   * `max_new_tokens` 控制最多生成多少
   * stop words / stop sequences 控制遇到某些模式停止
   * EOS token 是模型内部自然结束标记

7. **Beam Search 在开放生成中不一定最好**

   * 机器翻译等任务常用 beam search
   * 但在开放问答、聊天场景里，beam search 往往会让输出更模板化、更无聊

8. **参数调优没有通用最优值**

   * 事实性问答：通常偏低温、偏稳
   * 创作型任务：通常允许更高温度
   * 工具调用、代码生成：通常更保守

### 总结：一句话记忆

**解码参数不是改变模型知识，而是在控制模型如何从已有概率分布里做选择。**
