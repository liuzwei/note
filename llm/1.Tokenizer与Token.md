### 解释：对问题进行规范化解释

大模型并不是直接理解“文字”，而是先把文本切成一个个 **token**，再把 token 映射成向量。完成这一步的组件就是 **Tokenizer**。

所以你必须先建立一个非常稳固的认知：

**模型处理的基本单位通常不是字、不是词，而是 token。**

这件事会直接影响：

* 上下文长度
* API 计费
* 截断行为
* 训练样本组织
* 输出质量

---

### 举例：举例说明问题，帮助理解 token 不是“字数”

例如这几段文本：

* `hello`
* `ChatGPT`
* `中华人民共和国`
* `transformers`

它们在不同 tokenizer 里切分结果可能完全不一样：

* 有的按子词切
* 有的按字节切
* 有的把高频词当成一个 token
* 有的中文一个字可能是一个 token，也可能多个字组成一个 token

所以：

**1 个中文句子不等于固定数量 token，1 个英文单词也不等于 1 个 token。**

### 知识点：着重说明必须记住的技术点

1. **Tokenizer 的本质**

   * 它是“文本 <-> token id”之间的双向映射器
   * 包括：
   * 切分规则
   * 词表（Vocabulary）
   * 特殊 token 定义

2. **常见切分方式**

   * **BPE（Byte Pair Encoding）**
   * **WordPiece**
   * **Unigram / SentencePiece**
   * **Byte-level BPE**
   * 当前很多模型会采用 SentencePiece 或 byte-level 路线

3. **为什么不用纯按词切**

   * 未登录词太多
   * 词表会极大
   * 多语言泛化差
   * 子词切分可以兼顾词表大小和泛化能力

4. **特殊 token 必须知道**

   * `BOS`：序列开始
   * `EOS`：序列结束
   * `PAD`：补齐
   * `UNK`：未知 token
   * 特定聊天模型还会有角色分隔 token、工具调用 token 等

5. **token 数量决定成本**

   * 训练时：token 数基本就是计算量的重要尺度
   * 推理时：prefill 成本和输入 token 数强相关
   * API 使用时：通常也是按 token 计费

6. **不同模型 tokenizer 不兼容**

   * 同一句话，在不同模型中 token ids 可能完全不同
   * 这意味着模型权重和 tokenizer 是强绑定的
   * 不能随便替换 tokenizer 而不重训或适配

7. **训练数据处理的关键问题**

   * 截断策略
   * 拼接 packing
   * label 对齐
   * 特殊 token 注入
   * chat template 渲染后再 tokenize，而不是反过来

### 总结：一句话记忆

**Tokenizer 决定了模型看见世界的最小文本单位，token 不是字数统计，而是模型计算和成本的基本尺度。**
