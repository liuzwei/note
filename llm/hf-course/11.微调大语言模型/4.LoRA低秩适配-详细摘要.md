## LoRA 是什么

大语言模型的全量微调非常消耗资源。LoRA（Low-Rank Adaptation）是一种参数高效微调方法，它不去更新整个模型的全部权重，而是在原有权重之外，引入一组更小的可训练矩阵，并只训练这部分新增参数。

这种做法的直接效果是：

- 需要训练的参数量大幅下降
- 显存占用明显减少
- 更容易在消费级 GPU 上完成微调

课程页给出的直观结论是：LoRA 往往可以把可训练参数减少约 90%，而在更大的模型上，这种节省会更明显。

## 理解 LoRA 的核心思路

LoRA 的基本思想是：

1. 冻结预训练模型原有权重
2. 在模型层中注入低秩分解矩阵
3. 训练时只更新这些新增矩阵
4. 推理时可以把这部分权重合并回基础模型

它的关键点在于“低秩分解”。LoRA 假设：模型在适应新任务时，真正需要学习的权重更新量，并不一定需要一个完整的大矩阵，可以用两个更小的矩阵去近似表示。

这意味着：

- 参数更少
- 训练更轻
- 性能仍然可以保持在较好水平

原文给了一个典型例子：在 GPT-3 175B 上，LoRA 相比全量微调可把可训练参数降低到原来的万分之一量级，并把 GPU 显存需求降到约三分之一。

## LoRA 的主要优点

### 1. 显存效率高

LoRA 只需要把 adapter 参数放到 GPU 中参与训练，而基础模型参数可以保持冻结，甚至可以用更低精度加载。

这带来的实际意义是：

- 大模型微调门槛更低
- 对显存要求更友好
- 更适合实验和快速迭代

### 2. 训练集成简单

课程中强调，LoRA 已经和 PEFT、TRL 这些 Hugging Face 常用工具打通，因此实现层面非常直接。

你通常只需要：

- 定义 LoRA 配置
- 把配置传给 trainer
- 正常开始训练

此外，它还天然支持 QLoRA 这类更进一步节省显存的方案。

### 3. 便于 Adapter 管理

LoRA 不要求每次都保存整个微调后的模型。你可以只保存 adapter 权重。

这样做的好处是：

- 检查点更小
- 同一个基础模型可以挂多个 adapter
- 便于按任务切换不同适配器
- 部署时也可以再选择是否合并回基础模型

## 使用 PEFT 加载 LoRA Adapter

文章接着介绍了 Hugging Face 的 `PEFT` 库。这个库提供了统一接口，用来加载和管理包括 LoRA 在内的参数高效微调方法。

PEFT 的价值主要体现在：

- 可以把 adapter 加载到已有基础模型上
- 可以切换不同 adapter
- 不需要每次都重新训练完整模型

原文示例展示了一个常见流程：

```python
from peft import PeftModel, PeftConfig
config = PeftConfig.from_pretrained("ybelkada/opt-350m-lora")
model = AutoModelForCausalLM.from_pretrained(config.base_model_name_or_path)
lora_model = PeftModel.from_pretrained(model, "ybelkada/opt-350m-lora")
```

这个过程可以概括为：

1. 先读取 adapter 的配置
2. 根据配置加载基础模型
3. 再把 LoRA adapter 权重挂到基础模型上

如果你需要在多个任务适配器之间切换，PEFT 提供的接口会比手工处理权重方便得多。

## 使用 TRL + SFTTrainer 结合 LoRA 微调

课程页随后说明：如果你前一节已经用过 `SFTTrainer`，那么切换到 LoRA 版微调并不复杂。核心区别只是额外引入一份 `LoraConfig`。

整体流程可以概括成三步：

1. 定义 LoRA 参数
2. 创建带 `peft_config` 的 `SFTTrainer`
3. 训练并保存 adapter 权重

也就是说，LoRA 并不会彻底改变你的 SFT 工作流，而是在原有流程上做轻量增强。

## LoRA 配置中的关键参数

文章列出了几个最重要的 LoRA 参数。

### `r`

表示低秩矩阵的秩，也就是 LoRA 更新矩阵的内部维度。通常在 4 到 32 之间取值。

可以这样理解：

- `r` 越小，压缩越强，参数越少
- `r` 越大，表达能力越强，但显存和参数开销也会上升

### `lora_alpha`

这是 LoRA 层的缩放因子，通常会设置成 rank 的 2 倍左右。

它影响的是适配更新的强度：

- 值更高，适配效果更强
- 值过大，也可能增加训练不稳定或过拟合风险

### `lora_dropout`

这是 LoRA 层使用的 dropout 比例，常见值在 0.05 到 0.1 之间。

作用主要是：

- 降低过拟合风险
- 提升训练稳健性

### `bias`

控制是否训练 bias 项。常见选项包括：

- `none`
- `all`
- `lora_only`

从显存效率角度看，`none` 是最常见的配置。

### `target_modules`

指定把 LoRA 应用到哪些模块上。

可以是：

- `all-linear`
- 指定某些线性层，例如 `q_proj,v_proj`

一般来说：

- 覆盖更多模块，适应能力更强
- 但显存和训练成本也会相应上升

原文建议：先从较小的 rank 开始，例如 4 到 8，同时观察训练损失和验证效果，再逐步调整。

## 在 TRL 中使用 PEFT

课程页给出了一个 `LoraConfig` 配置示例，思路很直接：

```python
from peft import LoraConfig

rank_dimension = 6
lora_alpha = 8
lora_dropout = 0.05

peft_config = LoraConfig(
    r=rank_dimension,
    lora_alpha=lora_alpha,
    lora_dropout=lora_dropout,
    bias="none",
    target_modules="all-linear",
    task_type="CAUSAL_LM",
)
```

这段代码表达的重点是：

- 设定 LoRA 的秩
- 设定缩放因子
- 设定 dropout
- 指定作用模块
- 告诉框架当前任务是因果语言建模

随后，把这份配置传入 `SFTTrainer`：

```python
trainer = SFTTrainer(
    model=model,
    args=args,
    train_dataset=dataset["train"],
    peft_config=peft_config,
    max_seq_length=max_seq_length,
    processing_class=tokenizer,
)
```

这样 trainer 就会在训练过程中只更新 LoRA adapter，而不是整个模型参数。

原文还提到，模型加载时可以使用 `device_map="auto"` 自动分配设备，也可以手动指定设备。

## 合并 LoRA Adapter

LoRA 训练完成后，有时你不想在部署时再单独加载 adapter，而是希望得到一个已经和基础模型合并好的完整模型。

这时就需要做 merge。

合并的价值在于：

- 部署更简单
- 推理时不需要额外管理 adapter
- 最终只保留一个整合后的模型目录

但课程页也提醒了两个关键问题：

### 1. 内存管理

合并时通常需要同时加载：

- 基础模型
- adapter 权重

因此你要确认 GPU 或 CPU 内存足够，否则很容易在合并阶段爆内存。

### 2. 精度一致性

如果训练和保存使用的是 `float16`，那么在加载和合并时最好也保持相同精度。

这样可以避免：

- 不必要的类型转换
- 额外内存开销
- 保存结果与训练时不一致

## 合并实现示例

课程页给出的核心代码如下：

```python
import torch
from transformers import AutoModelForCausalLM
from peft import PeftModel

base_model = AutoModelForCausalLM.from_pretrained(
    "base_model_name", torch_dtype=torch.float16, device_map="auto"
)

peft_model = PeftModel.from_pretrained(
    base_model, "path/to/adapter", torch_dtype=torch.float16
)

merged_model = peft_model.merge_and_unload()
```

这段代码的执行步骤就是：

1. 加载基础模型
2. 加载 LoRA adapter
3. 调用 `merge_and_unload()` 把 adapter 权重合并回基础模型

如果你发现保存后的模型大小和预期不一致，原文建议同时保存 tokenizer：

```python
tokenizer = AutoTokenizer.from_pretrained("base_model_name")
merged_model.save_pretrained("path/to/save/merged_model")
tokenizer.save_pretrained("path/to/save/merged_model")
```

这一步能让部署目录更完整，避免后续加载时出现 tokenizer 缺失或配置不一致的问题。

## 本页的实践重点

这篇课程页真正想传达的重点，不只是“LoRA 是什么”，而是下面这组工程结论：

1. LoRA 是参数高效微调的主流方案之一
2. 它能显著降低大模型微调的资源成本
3. 它和 Hugging Face 的 PEFT、TRL 工具链结合得很好
4. 训练后可以保留 adapter，也可以再合并回基础模型
5. rank、alpha、dropout、target_modules 是最关键的调参入口
