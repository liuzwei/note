## 为什么微调后还要做评估

无论你是通过 SFT 还是 LoRA SFT 对模型做了微调，训练完成后都不能只看训练损失，还需要在标准基准和自定义场景上系统评估模型。

课程页强调的核心观点是：

- 机器学习工程师应该维护一套与目标领域相关的评估体系
- 标准 benchmark 可以帮助你横向比较模型
- 但 benchmark 分数本身并不等同于真实业务效果

也就是说，评估不是一个单点动作，而是一套持续维护的流程。

## 自动化基准测试是什么

自动化 benchmark 是评估语言模型能力的标准化工具。它们通常由三部分组成：

1. 精心整理的数据集
2. 明确规定的任务
3. 固定的评价指标

它们的主要优点是：

- 结果可复现
- 不同模型之间可直接比较
- 易于批量化执行

但课程同时提醒，这类基准只能作为完整评估体系中的一部分。  
一个模型即使在学术 benchmark 上表现很好，也不代表它一定适合你的真实业务场景。

## 如何理解自动化基准的价值与局限

自动化 benchmark 的价值在于“标准化”。

这意味着：

- 你能用同一套题目比较不同模型
- 你能稳定复现实验结果
- 你能快速判断模型在某些通用能力上的相对强弱

但它的局限同样明显：

- 很多 benchmark 距离真实使用环境较远
- 无法完全覆盖领域知识深度
- 不一定能反映复杂对话和长上下文交互能力
- 也不一定能体现实际部署中的错误代价

课程页想表达的是：**benchmark 很有用，但不能迷信 benchmark。**

## 常见通用基准

### 1. MMLU

MMLU 全称是 Massive Multitask Language Understanding，用来测试模型在 57 个学科上的知识能力，覆盖科学、人文等多个领域。

它适合用来观察模型的广泛知识掌握情况，但局限是：

- 更偏广度，不一定体现专业深度
- 对某些垂直领域来说不够细

### 2. TruthfulQA

TruthfulQA 用来评估模型是否容易复现常见误解或错误认知。

它的意义在于：

- 可以观察模型是否倾向于输出“听起来像对的，但其实是错的”内容

但它也不能覆盖所有形式的错误信息和幻觉问题。

## 推理类基准

### 1. BBH

BBH 是 Big Bench Hard，聚焦较难的推理任务。

它主要考察：

- 逻辑思维
- 规划能力
- 多步推理能力

### 2. GSM8K

GSM8K 主要用于评估数学文字题求解能力。

它更关注：

- 数学推导
- 多步计算
- 结构化求解过程

课程页的观点是，这类 benchmark 能帮助评估分析能力，但真实世界中的推理常常更复杂、更含糊，也更依赖上下文，所以 benchmark 分数只能反映其中一部分能力。

## 语言理解类评估

课程页提到了 HELM，它提供一个较全面的评估框架。

这类评估通常会从以下角度审视模型：

- 常识
- 世界知识
- 推理能力
- 语言处理能力

它的优势是视角更整体，但仍然很难完整覆盖自然对话中的复杂性，以及具体行业里的专业术语和任务要求。

## 领域专项基准

课程页特别提醒：如果你的模型服务于具体领域，那你不能只看通用 benchmark，还需要关注更有针对性的任务。

文中举了三个方向：

### 1. 数学领域：MATH

MATH benchmark 含有 12500 道数学竞赛题，覆盖：

- 代数
- 几何
- 数论
- 计数
- 概率等

它之所以难，是因为它要求模型具备：

- 多步推理能力
- 数学符号理解能力
- 分步骤生成解答的能力

### 2. 代码领域：HumanEval

HumanEval 是面向代码生成的 benchmark，包含 164 个编程题。

它的关键特点是：

- 不是只比较生成文本像不像参考答案
- 而是通过真实测试用例去检查代码功能是否正确

因此，它更能反映代码模型的实际可用性。

### 3. 对话与指令遵循：Alpaca Eval

Alpaca Eval 是自动化评估指令跟随模型质量的框架。  
它使用 GPT-4 作为 judge，从多个维度评估模型输出，比如：

- helpfulness
- honesty
- harmlessness

它的价值在于：

- 不依赖大量人工标注
- 评估可扩展
- 可以捕捉一些传统指标不容易反映的细粒度差异

## 替代性评估方法

除了标准 benchmark，很多组织还会用其他方式来补足评估体系。

### 1. LLM-as-Judge

这是一种“用一个大模型评估另一个大模型输出”的方法。

它的优点是：

- 比简单自动指标更细腻
- 能评估帮助性、表达质量、结构完整性等更复杂特征

它的缺点是：

- 评审模型本身也有偏差
- 不同 judge 模型可能给出不同结论

### 2. Evaluation Arenas

课程页提到了像 Chatbot Arena 这样的评估平台。

这类平台通常采用匿名对战机制：

1. 用户向两个模型提同一个问题
2. 用户不知模型身份
3. 用户投票选择更好的回答

这种方式的优势是：

- 更接近真实用户交互
- 可以收集大量多样化问题
- 与专家评估往往有较高一致性

但它也有局限：

- 用户群体有偏差
- Prompt 分布可能失衡
- 重点往往偏向“有帮助”，不一定充分覆盖安全性

### 3. 自定义内部基准

很多组织都会开发自己的 benchmark 套件，用来贴合自身业务和部署环境。

这类内部评估可能包括：

- 领域知识测试
- 特殊格式输出检查
- 模拟真实线上场景的问题集
- 高风险错误用例

## 如何建立更完整的评估策略

课程页给出了一套更实用的思路，不要只做标准 benchmark，而要做多层次评估。

可以概括为四步：

1. 先用相关标准 benchmark 建立基线，并与其他模型比较
2. 明确你真实业务中的关键任务和最不能接受的错误类型
3. 构建能反映真实场景的自定义评估数据集
4. 设计分层评估机制

这套分层机制通常包括：

- 自动化指标，用于快速反馈
- 人工评估，用于理解细节质量
- 领域专家评审，用于专业任务
- 受控环境中的 A/B 测试，用于上线前验证

这是本页最重要的工程结论之一：**真正有价值的评估，必须和实际场景绑定。**

## 如何做自定义评估

课程中建议，在设计自定义评估集时，至少要包含这些内容：

- 领域内的真实用户问题
- 经常出现的边界情况
- 特别困难或高风险的样本

这样做的目的，是让评估更贴近你最终部署时会面对的问题，而不是只在标准公开题集上追求分数。

## 使用 LightEval 做自动化评估

在实现层面，课程页推荐使用 `lighteval` 来评估微调后的模型。

它的优点是：

- 内置大量 benchmark 任务
- 可以通过统一格式选择任务
- 便于批量执行标准化评估

### 任务格式

LightEval 使用如下格式定义任务：

```text
{suite}|{task}|{num_few_shot}|{auto_reduce}
```

各字段含义如下：

- `suite`：benchmark 套件名，例如 `mmlu`
- `task`：具体任务名，例如 `abstract_algebra`
- `num_few_shot`：few-shot 示例数量，`0` 表示 zero-shot
- `auto_reduce`：如果 prompt 太长，是否自动减少 few-shot 样本，取值一般为 `0` 或 `1`

示例：

```text
mmlu|abstract_algebra|0|0
```

这表示：在 MMLU 的抽象代数任务上，以 zero-shot 方式评估模型。

## 评估管线示例

课程页给了一个基于医学相关子任务的评估示例，使用 `lighteval` 和 VLLM 后端：

```bash
lighteval accelerate \
    "pretrained=your-model-name" \
    "mmlu|anatomy|0|0" \
    "mmlu|high_school_biology|0|0" \
    "mmlu|high_school_chemistry|0|0" \
    "mmlu|professional_medicine|0|0" \
    --max_samples 40 \
    --batch_size 1 \
    --output_path "./results" \
    --save_generations true
```

这个示例体现了几个要点：

- 可以只挑某个领域相关的子任务
- 可以限制采样数量快速验证
- 可以保存生成结果，方便后续人工复查

## 结果怎么读

LightEval 会输出表格化结果，通常包括：

- 任务名
- 指标名
- 指标值
- 标准误差

原文示例中展示的是准确率 `acc`。

你可以重点关注：

- 总体平均成绩
- 各子任务之间的差异
- 哪些领域任务明显偏弱

如果某些子任务分数特别低，说明模型在该领域子能力上还存在明显短板。

课程页还提到，LightEval 也提供 Python API。  
如果你希望把评估结果接入自己的分析流程，或者做更细粒度的结果处理，Python API 会更灵活。

更多细节可参考：

- LightEval 文档：<https://github.com/huggingface/lighteval>

## 本页的核心结论

这篇课的重点可以压缩成下面几条：

1. 微调完成后，评估是必须环节
2. 标准 benchmark 适合建立基线，但不能代替真实场景评估
3. 应该结合通用 benchmark、领域 benchmark、自定义数据集和人工评估
4. LLM-as-Judge、Arena、A/B 测试等方法可以补足传统 benchmark 的不足
5. `lighteval` 是一个适合快速搭建标准评估流程的工具

## 本地实践建议

如果你要把这一节真正落地到自己的项目里，可以按这个顺序做：

1. 先挑 2 到 4 个与你任务最相关的公开 benchmark
2. 再整理一批真实业务样本作为自定义评估集
3. 同时保留自动指标和人工抽样检查
4. 对高风险任务单独做专项测试
5. 把评估流程固化成可重复执行的脚本

这样你的模型评估体系才不是一次性的，而是可持续迭代的。
