# 11.3 监督微调

> 说明：本文为 Hugging Face LLM Course 第 11 章第 3 节原文翻译版。

> 许可证：Apache License 2.0

监督微调（Supervised Fine-Tuning，SFT）是一种主要用于让预训练语言模型学会遵循指令、进行对话以及使用特定输出格式的过程。虽然预训练模型已经具备很强的通用能力，但 SFT 可以帮助它们转变成更像助手的模型，使其更好地理解并响应用户提示。通常，这是通过在由人工编写的对话和指令数据集上进行训练来完成的。

本页提供了一个分步骤指南，展示如何使用 [`SFTTrainer`](https://huggingface.co/docs/trl/en/sft_trainer) 对 [`deepseek-ai/DeepSeek-R1-Distill-Qwen-1.5B`](https://huggingface.co/deepseek-ai/DeepSeek-R1-Distill-Qwen-1.5B) 模型进行微调。按照这些步骤，你可以让模型更有效地适配特定任务。

## 何时使用 SFT

在进入实现细节之前，首先要理解什么时候 SFT 才是你的项目的正确选择。

第一步，你应该先考虑：对于你的使用场景，是否只靠一个现成的、经过指令微调的模型，加上一组设计良好的提示词，就已经足够了。SFT 需要明显的计算资源和工程投入，因此只有在对现有模型进行提示工程仍然无法满足需求时，才值得继续做 SFT。

> 提示
>
> 只有在以下情况下才考虑 SFT：
>
> - 你需要比提示工程能达到的更高性能
> - 你有明确的使用场景，在这个场景下，使用大型通用模型的成本高于微调一个较小模型的成本
> - 你需要专门的输出格式，或者现有模型难以处理的领域知识

如果你判断确实需要 SFT，那么是否推进，主要取决于两个因素：

### 模板控制

SFT 可以精确控制模型输出的结构。

当你需要模型做到以下几点时，这一点尤其有价值：

1. 按特定的聊天模板格式生成回复
2. 遵循严格的输出模式
3. 在不同回复之间保持一致的风格

### 领域适配

当你处理的是专业领域任务时，SFT 可以通过以下方式让模型更符合该领域的要求：

1. 学习领域术语和概念
2. 遵守专业标准
3. 正确处理技术性问题
4. 遵循行业特定指南

> 提示
>
> 在开始 SFT 之前，评估你的场景是否需要：
>
> - 精确的输出格式
> - 领域专属知识
> - 一致的响应模式
> - 遵循特定规范
>
> 这一步评估将帮助你判断 SFT 是否真的是适合你需求的方法。

## 数据集准备

监督微调需要一个面向具体任务的数据集，并且这个数据集要按输入-输出对的形式组织。

每一对样本应包含：

1. 一个输入提示词
2. 期望的模型回复
3. 任何额外的上下文或元数据

训练数据的质量对微调是否成功至关重要。接下来要看的就是如何准备并验证你的数据集。

> 注：原文此处嵌入了 `HuggingFaceTB/smoltalk` 数据集的在线预览。

## 训练配置

微调是否成功，很大程度上取决于你是否选择了合适的训练参数。下面逐项说明这些重要参数及其配置方式。

`SFTTrainer` 的配置需要综合考虑多个控制训练过程的参数。下面逐项解释这些参数及其用途：

1. 训练时长参数

   - `num_train_epochs`：控制总训练轮数
   - `max_steps`：轮数的替代方案，用来设置最大训练步数
   - 更大的 epoch 数有利于学习，但也会增加过拟合风险

2. Batch 大小参数

   - `per_device_train_batch_size`：决定显存占用和训练稳定性
   - `gradient_accumulation_steps`：在不额外增加显存的前提下实现更大的有效 batch size
   - 更大的 batch 往往能提供更稳定的梯度，但也需要更多内存

3. 学习率参数

   - `learning_rate`：控制权重更新的步长
   - `warmup_ratio`：用于学习率预热的训练阶段比例
   - 过高会导致训练不稳定，过低会导致学习速度过慢

4. 监控参数

   - `logging_steps`：记录训练指标的频率
   - `eval_steps`：在验证集上做评估的频率
   - `save_steps`：保存模型检查点的频率

> 提示
>
> 建议从较保守的参数开始，然后根据监控结果逐步调整：
>
> - 初始先从 1 到 3 个 epoch 开始
> - 一开始使用较小的 batch size
> - 密切观察验证指标
> - 如果训练不稳定，就调整学习率

## 使用 TRL 实现

现在我们已经理解了关键组成部分，接下来就通过合适的验证与监控来实现训练。

我们将使用 Transformers Reinforcement Learning（TRL）库中的 `SFTTrainer` 类，它是构建在 `transformers` 库之上的。

下面是一个使用 TRL 库的完整示例：

```python
from datasets import load_dataset
from trl import SFTConfig, SFTTrainer
import torch

# Set device
device = "cuda" if torch.cuda.is_available() else "cpu"

# Load dataset
dataset = load_dataset("HuggingFaceTB/smoltalk", "all")

# Configure model and tokenizer
model_name = "HuggingFaceTB/SmolLM2-135M"
model = AutoModelForCausalLM.from_pretrained(pretrained_model_name_or_path=model_name).to(
    device
)
tokenizer = AutoTokenizer.from_pretrained(pretrained_model_name_or_path=model_name)

# Setup chat template
model, tokenizer = setup_chat_format(model=model, tokenizer=tokenizer)

# Configure trainer
training_args = SFTConfig(
    output_dir="./sft_output",
    max_steps=1000,
    per_device_train_batch_size=4,
    learning_rate=5e-5,
    logging_steps=10,
    save_steps=100,
    eval_strategy="steps",
    eval_steps=50,
)

# Initialize trainer
trainer = SFTTrainer(
    model=model,
    args=training_args,
    train_dataset=dataset["train"],
    eval_dataset=dataset["test"],
    processing_class=tokenizer,
)

# Start training
trainer.train()
```

> 提示
>
> 当你使用包含 `"messages"` 字段的数据集时（就像上面的例子），`SFTTrainer` 会自动应用模型的聊天模板，这个模板会从 Hub 中读取。
>
> 这意味着你不需要额外配置来处理聊天式对话，trainer 会根据模型预期的模板格式来组织这些消息。

## 对数据集进行 Packing

`SFTTrainer` 支持样本打包（example packing），用来优化训练效率。  
这个特性允许把多个较短样本打包进同一条输入序列，从而在训练时最大化 GPU 利用率。

要启用 packing，只需要在 `SFTConfig` 构造函数中设置 `packing=True`。

当你在启用了 packing 的数据集上同时使用 `max_steps` 时，要注意：实际训练的 epoch 数可能会比你预期的更多，这取决于你的 packing 配置。

你可以通过一个格式化函数来自定义样本的组合方式。这在处理具有多个字段的数据集时尤其有用，例如问答对。对于评估数据集，你可以通过在 `SFTConfig` 中设置 `eval_packing=False` 来关闭 packing。

下面是一个自定义 packing 配置的基础示例：

```python
# Configure packing
training_args = SFTConfig(packing=True)

trainer = SFTTrainer(model=model, train_dataset=dataset, args=training_args)

trainer.train()
```

当你对包含多个字段的数据集做 packing 时，可以定义一个自定义格式化函数，把这些字段组合成一条输入序列。这个函数应当接收一个样本列表，并返回一个包含打包后输入序列的字典。

下面是一个自定义格式化函数的示例：

```python
def formatting_func(example):
    text = f"### Question: {example['question']}\n ### Answer: {example['answer']}"
    return text

training_args = SFTConfig(packing=True)
trainer = SFTTrainer(
    "facebook/opt-350m",
    train_dataset=dataset,
    args=training_args,
    formatting_func=formatting_func,
)
```

## 监控训练进度

有效监控对于成功完成微调至关重要。下面来看训练过程中应该重点关注什么。

### 理解损失曲线模式

训练损失通常会经历三个明显阶段：

1. 初始快速下降：模型迅速适应新的数据分布
2. 逐渐稳定：随着模型进一步微调，学习速度放缓
3. 收敛：损失值趋于稳定，表示训练接近完成

> 注：原文此处配有一张典型训练损失曲线示意图。

### 需要监控的指标

有效监控既包括跟踪定量指标，也包括评估定性指标。可用的指标包括：

- 训练损失
- 验证损失
- 学习率变化过程
- 梯度范数

> 警告
>
> 训练中要注意以下预警信号：
>
> 1. 训练损失下降，但验证损失上升（过拟合）
> 2. 损失值几乎没有明显改善（欠拟合）
> 3. 损失值异常低（可能发生记忆化）
> 4. 输出格式不一致（模板学习存在问题）

### 通往收敛的路径

随着训练推进，损失曲线应该逐步稳定下来。  
健康训练的关键标志是训练损失和验证损失之间的差距较小，这说明模型学到的是可以泛化的模式，而不是死记具体样本。

损失的绝对数值会因任务和数据集而不同。

### 训练进度监控

上面的图展示了一个典型的训练过程。可以注意到，训练损失和验证损失在一开始都快速下降，随后逐渐趋于平缓。这个模式表明模型正在有效学习，同时保持泛化能力。

### 需要警惕的异常信号

损失曲线中的若干模式可能意味着潜在问题。下面给出几个常见预警信号，以及可以考虑的应对方法。

> 注：原文此处依次给出了几张常见异常损失曲线配图。

如果验证损失的下降速度明显慢于训练损失，那么模型很可能正在对训练数据发生过拟合。

可以考虑：

- 减少训练步数
- 增加数据集规模
- 检查数据集质量和多样性

如果损失没有明显改善，模型可能存在以下问题：

- 学习过慢（可以尝试提高学习率）
- 任务过难（检查数据质量和任务复杂度）
- 已经触及模型架构能力上限（考虑更换模型）

极低的损失值可能意味着模型在“记忆”而不是真正“学习”。

如果出现以下情况，这尤其值得警惕：

- 模型在新的、相似的样本上表现很差
- 输出缺乏多样性
- 回复与训练样本过于相似

> 警告
>
> 训练期间既要看损失值，也要看模型的实际输出。有时损失看上去不错，但模型可能已经发展出不希望出现的行为。定期对模型输出进行定性评估，有助于捕捉那些单靠指标看不出来的问题。

需要说明的是，这里对损失值的解读主要针对最常见的情况。实际上，损失值的表现形式会因为模型、数据集、训练参数等因素而不同。如果你想进一步了解这些模式，可以阅读 [Fast AI 团队的这篇博客文章](https://www.fast.ai/posts/2023-09-04-learning-jumps/)。

## 测验

### 1. 在 SFT 中，哪些参数控制训练时长？

- `num_train_epochs` 和 `max_steps`
  解释：正确。这两个参数决定模型训练多久，一个按 epoch 数控制，一个按总 step 数控制。
- `batch_size` 和 `learning_rate`
  解释：它们会影响训练过程，但并不直接控制训练时长。
- `gradient_checkpointing` 和 `warmup_ratio`
  解释：这些参数影响训练效率和稳定性，而不是时长。

### 2. 损失曲线中的哪种模式表明可能出现了过拟合？

- 验证损失上升，而训练损失继续下降
  解释：正确。训练损失和验证损失之间出现这种分叉，是过拟合的经典信号。
- 训练损失和验证损失都稳定下降
  解释：这种模式实际上表示训练比较健康。
- 训练损失保持不变，而验证损失下降
  解释：这是一种不常见的模式，并不表示过拟合。

### 3. `gradient_accumulation_steps` 是用来做什么的？

- 在不额外增加显存的情况下增大有效 batch size
  解释：正确。它会在多次前向传播之间累积梯度，然后再统一更新权重。
- 在训练期间保存检查点
  解释：这由 `save_steps` 和 `save_strategy` 等参数控制。
- 控制学习率调度
  解释：学习率调度由 `learning_rate` 和 `warmup_ratio` 等参数控制。

### 4. 在 SFT 训练期间，你应该监控什么？

- 同时监控定量指标和定性输出
  解释：正确。两类监控都需要，才能尽可能发现各种潜在问题。
- 只监控训练损失
  解释：只看训练损失不足以保证模型行为良好。
- 只监控模型输出质量
  解释：虽然很重要，但如果只看定性结果，会错过训练过程中的很多关键信号。

### 5. 什么表明训练过程中的收敛是健康的？

- 训练损失和验证损失之间的差距较小
  解释：正确。这表示模型学到的是可以泛化的模式。
- 训练损失降到 0
  解释：极低的损失值可能意味着记忆化，而不是真正学会了任务。
- 验证损失低于训练损失
  解释：这通常并不常见，也可能意味着验证集存在问题。
