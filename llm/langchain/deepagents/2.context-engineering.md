# Deep Agents 中的上下文工程（Context Engineering）

> 以正确的格式提供正确的信息和工具，使你的 Deep Agent 能够可靠地完成任务。

---

上下文工程是以正确的格式提供正确的信息和工具，使你的 Deep Agent 能够可靠地完成任务。Deep Agent 可以访问多种上下文。有些来源在启动时提供给 Agent，另一些则会在运行时（例如用户输入）变得可用。Deep Agent 包含内置机制，用于管理跨长时间运行的会话的上下文。

本页概述了你的 Deep Agent 可以访问和管理的不同类型的上下文。

刚接触上下文工程？请参阅[概念概述](/oss/python/langgraph/context/overview)了解不同类型的上下文及其使用时机。

---

## 上下文类型

| 上下文类型 | 你控制的内容 | 作用域 |
|-----------|-------------|-------|
| **输入上下文** | Agent 启动时放入其提示的内容（系统提示、记忆、技能） | 静态，每次运行都应用 |
| **运行时上下文** | 调用时传入的静态配置（用户元数据、API 密钥、连接信息） | 每次运行，传播给子 Agent |
| **上下文压缩** | 内置的卸载（offloading）和摘要（summarization），将上下文保持在窗口限制内 | 自动，接近限制时触发 |
| **上下文隔离** | 使用子 Agent 隔离繁重工作，仅向主 Agent 返回结果 | 每个子 Agent，委派时启用 |
| **长期记忆** | 使用虚拟文件系统实现跨线程的持久化存储 | 跨对话持久化 |

---

## 输入上下文（Input context）

输入上下文是在启动时提供给 Deep Agent 的信息，这些信息会成为其系统提示的一部分。最终的提示由以下几个来源组成：

- **系统提示（System prompt）** — 你提供的自定义指令加上内置的 Agent 引导
- **记忆（Memory）** — 配置后始终加载的持久化 `AGENTS.md` 文件
- **技能（Skills）** — 根据相关性按需加载的能力（渐进式披露）
- **工具提示（Tool prompts）** — 使用内置工具或自定义工具的指令

---

### 系统提示

你的自定义系统提示会附加到内置系统提示之前，内置提示包含用于规划、文件系统工具和子 Agent 的引导。使用系统提示来定义 Agent 的角色、行为和知识：

```python
from deepagents import create_deep_agent

agent = create_deep_agent(
    model="google_genai:gemini-3.1-pro-preview",
    system_prompt=(
        "你是一名专注于科学文献的研究助手。"
        "始终引用来源。对于不同主题的并行研究，请使用子 Agent。"
    ),
)
```

`system_prompt` 参数是静态的，这意味着它不会随每次调用而改变。某些情况下你可能需要动态提示：例如，告诉模型"你拥有管理员权限"vs"你只有只读权限"，或从[长期记忆](#long-term-memory)注入"用户偏好简洁回复"等用户偏好。

如果你的提示依赖于上下文或 `runtime.store`，请使用 `@dynamic_prompt` 来构建上下文感知的指令。你的中间件可以读取 `request.runtime.context` 和 `request.runtime.store`。有关添加自定义中间件的信息，请参阅[定制指南](/oss/python/deepagents/customization)，LangChain 上下文工程指南中也有相关示例。

当工具单独使用上下文或 `runtime.store` 时，你**不需要**中间件；工具直接接收 `ToolRuntime` 对象（包括 `runtime.context` 和 `runtime.store`）。仅当工具需要与系统提示的更新一起打包时，才添加中间件。

要为特定提供商或模型调整组装后的系统提示，请使用[操控配置文件（harness profile）](/oss/python/deepagents/harness#profiles)：`base_system_prompt` 直接替换基础提示，`system_prompt_suffix` 追加到其后。

---

### 记忆（Memory）

记忆文件（`AGENTS.md`）提供持久化的上下文，始终加载到系统提示中。使用记忆来存储项目约定、用户偏好以及适用于每次对话的关键准则：

```python
agent = create_deep_agent(
    model="google_genai:gemini-3.1-pro-preview",
    memory=[
        "/project/AGENTS.md",
        "~/.deepagents/preferences.md",
    ],
)
```

与技能不同，记忆始终被注入——没有渐进式披露。保持记忆最小化以避免上下文过载；详细的流程和领域特定内容请使用技能。

参见[记忆](/oss/python/deepagents/memory)了解配置细节。

---

### 技能（Skills）

技能提供按需能力。Agent 在启动时读取每个 `SKILL.md` 的前置元数据（frontmatter），仅当确定技能相关时才会加载完整的技能内容。这既减少了 Token 用量，又提供专门的工作流程：

```python
agent = create_deep_agent(
    model="google_genai:gemini-3.1-pro-preview",
    skills=[
        "/skills/research/",
        "/skills/web-search/",
    ],
)
```

每个技能应专注于单一工作流程或领域；宽泛或重叠的技能会稀释相关性，并在加载时使上下文膨胀。在技能中，保持主要内容简洁，将详细的参考资料移至单独的文件，并在技能文件中引用。始终相关的约定请放入[记忆](#memory)。

参见[技能](/oss/python/deepagents/skills)了解编写和配置。

---

### 工具提示

工具提示是塑造模型如何使用工具的指令。所有工具都会暴露元数据（通常是模式和描述），模型可以在其提示中看到这些信息。

- **内置工具** —— 为 Deep Agent 添加操控能力（规划、文件系统、子 Agent）的中间件会自动将工具特定的指令附加到系统提示中，创建解释如何有效使用这些工具的工具提示：
  - **规划提示** —— `write_todos` 维护结构化任务列表的指令
  - **文件系统提示** —— `ls`、`read_file`、`write_file`、`edit_file`、`glob`、`grep`（以及使用沙箱后端时的 `execute`）的文档
  - **子 Agent 提示** —— 使用 `task` 工具委派工作的引导
  - **人工介入提示** —— 在指定工具调用时暂停的用法（当设置了 `interrupt_on` 时）
  - **本地上下文提示** —— 当前目录和项目信息（仅 CLI）
- **你提供的工具** —— 通过 `tools` 参数传入的工具，其描述（来自工具模式）会被发送给模型。你也可以添加自定义中间件来添加工具并附加自己的系统提示指令。

对于你提供的工具，请确保提供清晰的名称、描述和参数描述。这些信息引导模型推理何时以及如何使用该工具。在描述中说明何时使用该工具，并描述每个参数的作用：

```python
@tool(parse_docstring=True)
def search_orders(user_id: str, status: str, limit: int = 10) -> str:
    """按状态搜索用户订单。当用户询问订单历史或想检查订单状态时使用此工具。
    始终按提供的状态进行过滤。

    Args:
        user_id: 用户的唯一标识符
        status: 订单状态：'pending'、'shipped' 或 'delivered'
        limit: 返回结果的最大数量
    """
    # 实现代码...
```

要覆盖内置或用户提供工具的描述（针对特定提供商或模型），请使用[操控配置文件（harness profile）](/oss/python/deepagents/harness#profiles)的 `tool_description_overrides`，以工具名称作为键。`excluded_tools` 则从可见工具集中完全移除某个工具。

参见[操控](/oss/python/deepagents/harness)了解内置能力，[定制](/oss/python/deepagents/customization)了解如何直接传入工具。

---

### 完整系统提示

Deep Agent 的系统消息——即模型在运行开始时收到的组装好的系统提示——由以下部分组成：

1. 自定义 `system_prompt`（如果提供）
2. 基础 Agent 提示
3. 待办列表提示：如何使用待办列表规划的指令
4. 记忆提示：`AGENTS.md` + 记忆使用指南（仅在提供记忆时）
5. 技能提示：技能位置 + 技能列表及其前置元数据信息 + 用法（仅在提供技能时）
6. 虚拟文件系统提示（文件系统 + 可执行工具文档，如适用）
7. 子 Agent 提示：任务工具用法
8. 用户提供的中间件提示（如果提供了自定义中间件）
9. 人工介入提示（当设置了 `interrupt_on` 时）

## 运行时上下文（Runtime context）

运行时上下文是每次运行的配置，你在调用 Agent 时传入。它**不会**自动包含在模型提示中；只有当某个工具、中间件或其他逻辑读取它并将其添加到消息或系统提示时，模型才能看到它。

使用运行时上下文存储用户元数据（ID、偏好、角色）、API 密钥、数据库连接、功能标志或其他你的工具和操控所需的值。使用 `context_schema` 定义数据形状：使用 `dataclasses.dataclass` 或 `typing.TypedDict` 类。使用 `invoke`/`ainvoke` 的 `context` 参数传入值。

完整详情请参见[运行时](/oss/python/langgraph/agents#runtime)和 [LangGraph 运行时上下文](/oss/python/langgraph/context)。

在工具内部，从注入的 `ToolRuntime` 读取上下文：

```python
from dataclasses import dataclass
from deepagents import create_deep_agent
from langchain.tools import tool, ToolRuntime


@dataclass
class Context:
    user_id: str
    api_key: str


@tool
def fetch_user_data(query: str, runtime: ToolRuntime[Context]) -> str:
    """获取当前用户的数据。"""
    user_id = runtime.context.user_id
    return f"用户 {user_id} 的数据：{query}"


agent = create_deep_agent(
    model="google_genai:gemini-3.1-pro-preview",
    tools=[fetch_user_data],
    context_schema=Context,
)

result = agent.invoke(
    {"messages": [{"role": "user", "content": "获取我的最近活动"}]},
    context=Context(user_id="user-123", api_key="sk-..."),
)
```

运行时上下文会传播到所有**子 Agent**。当子 Agent 运行时，它接收与父 Agent 相同的运行时上下文。参见[子 Agent](/oss/python/deepagents/subagents)了解每个子 Agent 的上下文（带命名空间的键）。

## 上下文压缩（Context compression）

长时间运行的任务会产生大量的工具输出和长对话历史。上下文压缩可减少 Agent 工作记忆中的信息量，同时保留与任务相关的细节。以下是将传递给 LLM 的上下文保持在其上下文窗口限制内的内置机制。

### 卸载（Offloading）

Deep Agent 使用内置的文件系统工具自动卸载内容，并在需要时搜索和检索这些已卸载的内容。当工具调用的输入或结果超过 Token 阈值（默认 **20,000**）时，会发生内容卸载：

- **工具调用输入超过 20,000 Token**：文件写入和编辑操作会在 Agent 对话历史中留下包含完整文件内容的工具调用。由于这些内容已持久化到文件系统，通常是冗余的。当会话上下文超过模型可用窗口的 **85%** 时，Deep Agent 会截断较早的工具调用，将其替换为指向磁盘上文件的指针，从而缩减活跃上下文的大小。

- **工具调用结果超过 20,000 Token**：当发生这种情况时，Deep Agent 会将响应卸载到配置的后端，并用文件路径引用和前 10 行预览来替代。Agent 可以在需要时重新读取或搜索内容。

### 摘要（Summarization）

当上下文大小超过模型的上下文窗口限制（例如 `max_input_tokens` 的 85%），且没有更多可卸载的上下文时，Deep Agent 会对消息历史进行摘要。此过程包含两个部分：

- **上下文内摘要**：LLM 生成对话的结构化摘要，包括会话意图、创建的工作成果和后续步骤——该摘要将替换 Agent 工作记忆中的完整对话历史。
- **文件系统保留**：完整、原始的对话消息会写入文件系统作为规范记录。

这种双重方法确保 Agent 能够通过摘要保持对其目标和进度的感知，同时在需要时（通过文件系统搜索）保留恢复特定细节的能力。

**配置：**
- 触发阈值：模型配置文件中 `max_input_tokens` 的 85%
- 保留最新上下文的 10% Token
- 如果模型配置文件不可用，回退到 170,000 Token 触发 / 保留 6 条消息
- 如果任何模型调用引发标准的 `ContextOverflowError`，Deep Agent 立即回退到摘要，并用摘要 + 最近保留的消息重试
- 旧消息由模型进行摘要

Agent 流式输出的 Token 通常包含摘要步骤生成的 Token。你可以使用关联的元数据过滤掉这些 Token：

```python
for chunk in agent.stream(
    {"messages": [...]},
    stream_mode="messages",
    version="v2",
):
    token, metadata = chunk["data"]
    if metadata.get("lc_source") == "summarization":
        continue
    else:
        ...
```

### 摘要工具（Summarization Tool）

Deep Agent 包含一个可选的摘要工具，使 Agent 能够在合适的时机（例如任务之间）触发摘要，而不是在固定的 Token 间隔。你可以通过将摘要工具附加到中间件列表来启用它：

```python
from deepagents import create_deep_agent
from deepagents.backends import StateBackend
from deepagents.middleware.summarization import (
    create_summarization_tool_middleware,
)

backend = StateBackend  # 如果使用默认后端
model = "google_genai:gemini-3.1-pro-preview"

agent = create_deep_agent(
    model=model,
    middleware=[
        create_summarization_tool_middleware(model, backend),
    ],
)
```

启用此功能不会禁用默认的 85% 限制摘要操作。详情请参见 [SummarizationToolMiddleware API 参考](https://reference.langchain.com/python/deepagents/middleware/summarization/create_summarization_tool_middleware)。

## 使用子 Agent 实现上下文隔离

子 Agent 解决了**上下文膨胀**问题。当主 Agent 使用输出量大的工具（网络搜索、文件读取、数据库查询）时，上下文窗口会迅速填满。子 Agent 隔离这些工作——主 Agent 只接收最终结果，而不是产生结果的那几十个工具调用。你还可以单独配置每个子 Agent（例如模型、工具、系统提示和技能）。

**工作原理：**
1. 主 Agent 拥有一个用于委派工作的任务工具
2. 子 Agent 使用自己全新的上下文运行
3. 子 Agent 自主执行直到完成
4. 子 Agent 向主 Agent 返回一份最终报告
5. 主 Agent 的上下文保持整洁

**最佳实践：**

- **委派复杂任务**：使用子 Agent 处理会扰乱主 Agent 上下文的多步骤工作。
- **保持子 Agent 响应简洁**：指示子 Agent 返回摘要而非原始数据：

```python
research_subagent = {
    "name": "researcher",
    "description": "对某个主题进行研究",
    "system_prompt": (
        "你是一名研究助手。重要提示：仅返回必要的摘要（500 字以内）。"
        "不要包含原始的搜索结果或详细的工具输出。"
    ),
    "tools": [web_search],
}
```

- **对大数据使用文件系统**：子 Agent 可以将结果写入文件，主 Agent 读取所需内容。

参见[子 Agent](/oss/python/deepagents/subagents)了解配置和上下文管理，以及运行时上下文传播和每个子 Agent 的命名空间。

## 长期记忆（Long-term memory）

使用默认文件系统时，Deep Agent 的工作记忆文件存储在 Agent 状态中，仅在单个线程内持久化。长期记忆使 Deep Agent 能够在不同线程和对话之间持久化信息。

Deep Agent 可以使用长期记忆来存储用户偏好、积累的知识、研究进度或任何应在单个会话之外持久化的信息。

要使用长期记忆，你必须使用 `CompositeBackend`，将特定路径（通常是 `/memories/`）路由到 LangGraph Store，后者提供持久的跨线程持久化。`CompositeBackend` 是一个混合存储系统，某些文件无限期持久化，而其他文件则限于单个线程。

```python
from deepagents import create_deep_agent
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend
from langgraph.store.memory import InMemoryStore


def make_backend(runtime):
    return CompositeBackend(
        default=StateBackend(runtime),
        routes={
            "/memories/": StoreBackend(runtime),
        },
    )


agent = create_deep_agent(
    model="google_genai:gemini-3.1-pro-preview",
    store=InMemoryStore(),
    backend=make_backend,
    system_prompt=(
        "当用户告诉你他们的偏好时，"
        "将其保存到 /memories/user_preferences.txt，"
        "以便你在未来的对话中记住它们。"
    ),
)
```

你**不需要**预填充 `/memories/` 文件。你提供后端配置、存储（store）和系统提示指令，告诉 Agent **保存什么**以及**保存到哪里**。例如，你可以提示 Agent 将偏好存储在 `/memories/preferences.txt` 中。路径初始为空，当用户分享值得记住的信息时，Agent 会使用其文件系统工具（`write_file`、`edit_file`）按需创建文件。

要预植入记忆，请在 LangSmith 上部署时使用 Store API。参见[长期记忆](/oss/python/deepagents/memory)了解设置和使用案例。

## 最佳实践

| 实践 | 说明 |
|------|------|
| **从正确的输入上下文开始** | 保持记忆最小化，仅存放始终相关的约定；对特定任务的能力使用聚焦的技能 |
| **利用子 Agent 处理繁重工作** | 将多步骤、输出量大的任务委派出去，保持主 Agent 上下文的整洁 |
| **在配置中调整子 Agent 输出** | 如果调试时发现子 Agent 生成长输出，可以在子 Agent 的 `system_prompt` 中添加引导，要求其创建摘要和综合发现 |
| **使用文件系统** | 将大输出持久化到文件（例如子 Agent 写入或自动卸载），使活跃上下文保持小巧；模型在需要细节时可以用 `read_file` 和 `grep` 拉取片段 |
| **记录长期记忆结构** | 告诉 Agent `/memories/` 中存放了什么以及如何使用它们 |
| **为工具传递运行时上下文** | 使用上下文传递用户元数据、API 密钥以及工具所需的其他静态配置 |
